# Quick Task 260407-ls3: WindowedHFDroneDataset must return 32 kHz raw audio (not mel)

## Problem

`WindowedHFDroneDataset.__getitem__` ([src/acoustic/training/hf_dataset.py:253-298](src/acoustic/training/hf_dataset.py#L253)) currently returns a `(1, 128, 64)` mel spectrogram tensor. The Phase 20 v7 EfficientAT trainer ([src/acoustic/training/efficientat_trainer.py:597](src/acoustic/training/efficientat_trainer.py#L597)) feeds the dataloader output through `mel = mel_train(batch_wav)`, which calls `AugmentMelSTFT.forward()` whose first line is:

```python
x = nn.functional.conv1d(x.unsqueeze(1), self.preemphasis_coefficient).squeeze(1)
```

That `conv1d` expects raw waveform `(B, T)`. When fed a batched mel of shape `(64, 1, 128, 64)`, the `.unsqueeze(1)` produces `(64, 1, 1, 128, 64)` and conv1d crashes:

```
RuntimeError: Expected 2D (unbatched) or 3D (batched) input to conv1d,
but got input of size: [64, 1, 1, 128, 64]
```

This killed Vertex job `8506941650449203200` (Phase 20 v7, fix2 image) at 12:27:39 UTC, ~30 seconds into stage 1.

## Root cause

Plan 20-03 designed `WindowedHFDroneDataset` against the wrong contract for Phase 20. It mirrors the legacy `HFDroneDataset` (which returns mel features for the deprecated `ResearchCNN` path), but Phase 20 v7 is **exclusively** `efficientat_mn10` per `vertex_submit.py:build_env_vars_v7` (D-23). EfficientAT does its own mel + spec-aug transform inside the training loop via `AugmentMelSTFT(freqm, timem, …)`, so the dataset must hand it raw audio at 32 kHz — exactly the contract that `_LazyEfficientATDataset` ([efficientat_trainer.py:87-137](src/acoustic/training/efficientat_trainer.py#L87)) already implements.

Two consequences:
1. **Wrong shape:** dataset returns `(1, 128, 64)` mel; trainer needs `(T,)` raw waveform.
2. **Wrong sample rate:** dataset operates at 16 kHz throughout; EfficientAT needs 32 kHz.

The window math (`window_samples=8000` = 0.5 s @ 16 kHz) is correct as input parameters, but the slice itself must be resampled to 32 kHz (final length 16000) before being returned.

A third subtle pre-existing issue masked this: Plan 20-03's RED stub `tests/unit/test_sliding_window_dataset.py` uses an unrelated `file_lengths=` constructor API that doesn't match the real class, so it never executed against the real implementation. Plan 20-04's trainer-wiring tests didn't exercise the windowed-EfficientAT path end-to-end either. **Out of scope for this quick task** but flagged for future cleanup.

## Goal

Make `WindowedHFDroneDataset` produce (raw_audio_32k, label) pairs compatible with `mel_train(batch_wav)` so Phase 20 v7 training can complete on Vertex. Preserve all existing properties:
- Sliding-window enumeration (D-13/D-14)
- Session-level file isolation (D-15)
- Non-uniform clip tolerance from quick task `260407-jx8` (`pad_or_loop` fallback)
- Unique-file-indices assertion
- Per-file label cache
- One-shot warning on non-uniform clips

## Approach

Mirror `_LazyEfficientATDataset`'s contract inside `WindowedHFDroneDataset.__getitem__`:

1. Decode WAV bytes → 16 kHz numpy float32 (unchanged).
2. Tolerate non-uniform clips via the existing `pad_or_loop`/truncate path (unchanged).
3. Slice the deterministic window at 16 kHz: `audio[offset : offset + window_samples]` (unchanged; pad with zeros if tail too short).
4. Apply `waveform_aug` at 16 kHz BEFORE resampling — this is consistent with how the existing comment at line 274 says it should happen, and it's what `_build_train_augmentation()` augmentations expect.
5. **NEW:** Resample 16 kHz → 32 kHz via `torchaudio.functional.resample`. Window of 8000 samples @ 16 kHz becomes 16000 samples @ 32 kHz.
6. **NEW:** Return raw 1D `torch.Tensor` of length 16000, not a mel spectrogram.
7. **REMOVE:** Mel spectrogram computation (`mel_spectrogram_from_segment`) and its `(1, 128, 64)` shape.
8. **REMOVE:** `spec_aug` parameter and its application — EfficientAT's `AugmentMelSTFT(freqm=..., timem=...)` already handles spec-aug per-batch on device, and there's no other consumer.
9. **REMOVE:** `mel_config` parameter — no longer used by this class.
10. Drop the now-unused `MelConfig` import for `WindowedHFDroneDataset` (still needed by `HFDroneDataset` so the module-level import stays).

The `efficientat_trainer.py` constructor call sites at lines 439-452 already pass only the parameters that survive (`file_indices`, `window_samples`, `hop_samples`, `waveform_aug`). Removing `mel_config`/`spec_aug` from the signature is API-safe.

Update `tests/unit/test_windowed_dataset_non_uniform.py` to:
- Assert each `__getitem__` returns a 1D `torch.Tensor` of length 16000 (= 0.5 s × 32 kHz), `dtype=float32`.
- Remove the `(1, 128, 64)` shape check.
- Keep the non-uniform-clip warning assertion (still relevant — pad_or_loop path is unchanged).
- Add a sanity check that the returned tensor for a known input is finite.

## must_haves

- `WindowedHFDroneDataset.__getitem__` returns `(torch.Tensor[16000], torch.Tensor[scalar])` — raw audio at 32 kHz, not mel.
- Resampling is via `torchaudio.functional.resample(..., orig_freq=16000, new_freq=32000)` — consistent with `_LazyEfficientATDataset` line 120.
- `mel_config` and `spec_aug` parameters removed from `WindowedHFDroneDataset.__init__`.
- All existing call sites in `efficientat_trainer.py:439-452` continue to work without modification.
- Non-uniform-clip handling from quick task `260407-jx8` is preserved (pad_or_loop, one-shot warning).
- `tests/unit/test_windowed_dataset_non_uniform.py` updated and passes.
- No regressions: `tests/unit/test_sliding_window_dataset.py` was already broken (RED stub with wrong API) — keep it alone, do not touch.
