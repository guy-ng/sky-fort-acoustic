# Quick Task 260407-jx8: Fix WindowedHFDroneDataset assertion on non-uniform DADS clips

## Problem

`WindowedHFDroneDataset.__getitem__` (`src/acoustic/training/hf_dataset.py:256-264`) raises
`AssertionError` whenever a DADS clip's decoded length differs from
`assumed_clip_samples` (16000). Phase 20 v7 Vertex training crashed at
`file_idx=174151` because that file decodes to 8000 samples (0.5 s) instead of the
assumed 1 s. The "uniform-clip" assumption from 20-RESEARCH A1 is violated by at
least one outlier clip in the upstream dataset, and the loud failure path is
itself the bug â€” it kills training instead of degrading gracefully.

## Goal

Make `WindowedHFDroneDataset` tolerate clips whose decoded length is shorter or
longer than `assumed_clip_samples` so Phase 20 v7 training can complete without
losing batches. Sliding-window math is computed once at `__init__` from
`assumed_clip_samples`, so per-file `__getitem__` must reconcile the actual
audio against that assumption.

## Approach

Replace the hard `AssertionError` with a tolerant fallback that preserves the
pre-computed window count:

1. If `len(audio) < assumed_clip_samples`: tile the clip to `assumed_clip_samples`
   via the existing `pad_or_loop` helper, then slice the deterministic window.
   This keeps every pre-allocated `(file_idx, offset)` pair valid and produces
   meaningful (looped) audio rather than zero-padded silence.
2. If `len(audio) > assumed_clip_samples`: truncate to `assumed_clip_samples`
   so window offsets stay in-bounds.
3. Emit a `logger.warning` the **first time** a non-uniform clip is encountered
   (gated by an instance flag) so the operator sees the divergence without
   spamming logs across millions of `__getitem__` calls.
4. Preserve the existing tail-pad-with-zeros path at the end of the slice
   (handles the edge case of tail offsets in shorter clips after looping).

This fix is intentionally minimal â€” it does NOT recompute per-file window counts
at `__init__`, which would require decoding all 174k+ files up front. The
looping fallback is consistent with how `HFDroneDataset.__getitem__` already
handles short clips (`pad_or_loop` at line 76).

## must_haves

- `src/acoustic/training/hf_dataset.py` no longer raises `AssertionError`
  when `len(audio) != assumed_clip_samples`.
- Looping fallback (`pad_or_loop`) is used for short clips so windows contain
  real audio, not silence.
- A unit test in `tests/unit/` constructs a `WindowedHFDroneDataset` over a
  fake HF dataset that mixes uniform (16000-sample) and short (8000-sample)
  clips, iterates the full length, and asserts no exception is raised.
- The new test passes locally.
- A `logger.warning` fires once per dataset instance on the first non-uniform
  encounter.
